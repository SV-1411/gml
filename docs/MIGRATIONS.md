# Database Migrations Guide

Complete guide for using Alembic with async SQLAlchemy for database migrations.

## Quick Start

### 1. Install Dependencies

```bash
cd "/Volumes/Yatri Cloud/org/gml/project"
pip install -r requirements-dev.txt
```

### 2. Generate Initial Migration

Since Alembic is already initialized, generate the first migration for all existing models:

```bash
alembic revision --autogenerate -m "initial migration - create all tables"
```

This will create a migration file in `alembic/versions/` that includes:
- Agent table
- Message table
- Capability table
- Memory table
- Cost table
- AuditLog table
- Connection table

### 3. Review the Migration

**IMPORTANT**: Always review the generated migration file before applying it!

```bash
# View the generated migration file
cat alembic/versions/XXXX_initial_migration_create_all_tables.py
```

### 4. Apply the Migration

```bash
# Apply the migration to your database
alembic upgrade head
```

### 5. Verify Migration

```bash
# Check current database revision
alembic current

# View migration history
alembic history
```

## Common Commands

### Generate New Migration

```bash
# After making model changes, generate migration
alembic revision --autogenerate -m "add new field to agent table"
```

### Apply Migrations

```bash
# Apply all pending migrations
alembic upgrade head

# Apply one migration at a time
alembic upgrade +1

# Apply up to specific revision
alembic upgrade <revision_id>
```

### Rollback Migrations

```bash
# Rollback one migration
alembic downgrade -1

# Rollback to specific revision
alembic downgrade <revision_id>

# Rollback all migrations (⚠️ DANGEROUS)
alembic downgrade base
```

### View Status

```bash
# Show current database revision
alembic current

# Show all migrations
alembic history

# Show pending migrations
alembic heads

# Show what SQL will be executed (dry run)
alembic upgrade head --sql
```

## Migration Workflow

### Development Workflow

1. **Make model changes** in `src/gml/db/models.py`
2. **Generate migration**: `alembic revision --autogenerate -m "description"`
3. **Review migration file** in `alembic/versions/`
4. **Test migration**: `alembic upgrade head`
5. **Verify changes** in database
6. **Commit migration file** to version control

### Production Workflow

1. **Backup database** before migrations
2. **Review migration** carefully
3. **Test on staging** first
4. **Apply migration**: `alembic upgrade head`
5. **Verify application** works correctly
6. **Monitor** for any issues

## Rollback Procedures

### Safe Rollback (One Step)

```bash
# Check current revision first
alembic current

# Rollback one migration
alembic downgrade -1

# Verify rollback
alembic current
```

### Rollback to Specific Revision

```bash
# View history to find revision ID
alembic history

# Rollback to specific revision
alembic downgrade <revision_id>
```

### Emergency Full Rollback

```bash
# ⚠️ WARNING: This will drop all tables!
# Only use if you have backups and understand the consequences

alembic downgrade base
```

## Migration Naming Conventions

Use descriptive, lowercase names with underscores:

✅ **Good:**
- `initial_migration_create_all_tables`
- `add_index_to_agents_status`
- `modify_message_payload_type`
- `create_user_preferences_table`

❌ **Bad:**
- `migration1`
- `update`
- `fix`
- `changes`

## Troubleshooting

### Migration Conflicts

If you have multiple migration branches:

```bash
# Merge branches
alembic merge -m "merge feature branch" <rev1> <rev2>
```

### Check Migration SQL

Before applying, see what SQL will be executed:

```bash
# See upgrade SQL
alembic upgrade head --sql

# See downgrade SQL
alembic downgrade -1 --sql
```

### Database Connection Issues

Ensure your `.env` file has the correct `DATABASE_URL`:

```bash
# Check database URL format
# Should be: postgresql://user:password@host:port/database
# Or: postgresql+asyncpg://user:password@host:port/database
```

### Model Not Detected

If a model isn't detected in autogenerate:

1. Ensure model is imported in `alembic/env.py`
2. Ensure model inherits from `Base`
3. Check for syntax errors in model file

## Configuration

The Alembic configuration is in:
- `alembic.ini` - Main configuration file
- `alembic/env.py` - Environment setup with async support
- `alembic/script.py.mako` - Migration template

Database URL is automatically loaded from `src.gml.core.config.get_settings()`.

## Best Practices

1. ✅ **Always review** autogenerated migrations
2. ✅ **Test migrations** on development database first
3. ✅ **Create backups** before production migrations
4. ✅ **Use descriptive names** for migrations
5. ✅ **Keep migrations small** - one logical change per migration
6. ✅ **Never edit** existing migrations - create new ones
7. ✅ **Document complex migrations** with comments
8. ✅ **Test rollback** procedures in development

## Example: Complete Migration Cycle

```bash
# 1. Make model changes
# Edit src/gml/db/models.py

# 2. Generate migration
alembic revision --autogenerate -m "add email field to agent table"

# 3. Review generated file
cat alembic/versions/XXXX_add_email_field_to_agent_table.py

# 4. Apply migration
alembic upgrade head

# 5. Verify
alembic current

# 6. If something goes wrong, rollback
alembic downgrade -1

# 7. Fix and regenerate
alembic revision --autogenerate -m "add email field to agent table v2"

# 8. Apply again
alembic upgrade head
```
